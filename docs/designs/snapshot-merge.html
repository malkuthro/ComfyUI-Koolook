Snapshot Merge / Branch Review — Mockup

A "GitHub-style" branch + merge UX for snapshots. Two maintainers work from the same starting point, end up with diverging snapshots, then review and selectively merge each other's work — without ever overwriting silently. Discussion-only; no implementation. The storage layer behind this can be plain JSON files with a lineage_id + parent_hash (v1), a real git remote (v2), or a hosted service (v3) — this mockup is the UI surface, independent of where the bytes live. Schema referenced throughout matches the actual snapshot shape from web/sidebar/snapshot.js: {kind, version, name, exportedAt, picks: [], workflows: {directories: {}}}.

Add — present in theirs, not in yours Remove — present in yours, not in theirs Conflict — same item changed differently on both sides Theirs — the incoming branch

Adds default to checked; removes default to unchecked. Picking up new picks from Alice is the common case; deleting a pick you intentionally kept is the rare destructive case. Unchanged rows are non-interactive (dimmed) — purely to give the user the "and these are the 9 we both already have" reassurance.
Group toggles in the section header (All adds / All removes) cascade to the rows. The same checkbox / decision model carries into the workflows section below.

Adding a folder is a single tick that cascades. Checking Color science/ implicitly checks both child files. Unticking a child leaves the folder "partial" (visualized with a half-shaded checkbox in the live UI, not shown here). Folder removes would behave symmetrically — uncheck the folder to keep the whole tree on your side.
The leaf with the conflict (comp_v02.json) is not committable from this row alone — its checkbox is "armed" but the actual decision is taken in the conflict pane below. Until the user resolves it (or skips), the row stays amber.

"Keep both" is the safe default for conflicting workflows. Workflows are opaque (JSON node graphs); we can't safely line-merge two of them like text — a merged graph might run but produce nonsense. So the third option preserves both, with the incoming one renamed by suffix. The user can open both in the canvas afterwards and decide which to keep on their own time.
Picks conflicts don't exist — picks are a set of string IDs, so the only conflict is "in yours and not in theirs" or vice versa, which is exactly the add/remove rows above. The three-way choice only applies to workflow files.

Default = save into the current snapshot. The user opened this from their preset row, so "merge into mine" is the dominant intent. Save as new branch covers "I want to try Alice's flavor on top of mine without losing my pre-merge version" — picks a new name automatically (e.g. Wan22 VFX kit (merged Alice 05-14)) but the user can rename.
The pre-merge state is always preserved. Same as a regular Load, a pre_load_*.json autosave of the current state is written into the preset's _autosave/ folder first. If the merge result is wrong, the user can recover via the existing Recovery section (PR #144) — no new recovery surface needed.

8. Out of scope (for this sketch)

NotesDeliberately deferred — pin down before any code
  • Transport. How Alice's snapshot lands in your library — file paste, drag-drop, git pull, hosted-service fetch — is a separate decision. The dialog only needs a snapshot file with a matching lineage_id.
  • Identity / auth. "Alice" is the author field on the snapshot envelope. v1 trusts whatever the file says (same trust model as opening a workflow JSON today). Signed snapshots or a real account system is a v2+ concern.
  • Three-way merge of workflow internals. Out of scope on purpose — see §6 annotation. v1 ships the "keep both rename" affordance; a true graph-level merge tool is its own project.
  • Multi-branch merge in one pass. v1 supports one incoming branch at a time. Stacking Alice + Ben in one review is a v2 polish.
  • Live collaboration / CRDT. This is explicitly the "review and merge" pattern, not "Google Docs for snapshots." If real-time co-editing is desired, that's a different feature entirely (different storage, different UI).
  • History view. Showing the full lineage tree (Yours → Alice → merge → next branch) is a nice-to-have but not in the merge flow. Could live in a separate "snapshot history" surface later.
