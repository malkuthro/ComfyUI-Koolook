Each option is rendered as a normal preset-row — same shape the user already reads in the Load dialog (bold name + picks · workflows · saved-date meta line), so there's nothing new to parse. The two rows make the diff obvious without spelling it out: the autosave row carries a later timestamp and (typically) higher picks/workflow counts because it captured WIP added after the morning save. Buttons collapse to NO / YES — load latest so they sit on one row and read as a binary.

5. Button state pattern (consistent across every dialog)

PatternThe primary button's label, color, and disabled-ness reflect what the user is about to do

The post-PR-130 Settings dialog already does this for "Save folder" → "Saved" (modals.js:1916). Apply the same idea everywhere a primary action button exists, so the user always sees what the system thinks is happening.

Default
Clean state. Save writes over the currently tracked preset; nothing about the destination has changed.
Dirty — folder changed
User picked a different destination via "Save to…". Label spells out the consequence so the click isn't a surprise.
In progress
Write in flight. Button disables to prevent double-submit; color shifts to neutral-blue so it doesn't read "danger".
Done
After a successful write, briefly. Same pattern as the Settings dialog's idle "Saved" state.
Same four-state cycle applies to "Load from…", "Save to…", and any future primary action. Doesn't require new infrastructure — just discipline: every place a button is wired, write the four label/state branches up front.

6. Folder-browse picker — what opens when you click Save to… / Load from…

ConstraintWhy this isn't the OS-native picker

ComfyUI runs in a browser tab. Browsers don't expose the OS folder picker to JavaScript — <input type="file"> only picks files, and showDirectoryPicker() is Chromium-only with an aggressive permission prompt that grants ongoing read access. A server-side directory-listing route + in-app picker is the standard pattern when a web-served tool needs a folder path. PR #132 is already adding the practical mitigation: a path input where users can paste a path copied from Explorer's address bar / Finder's Cmd+Opt+C.

1
User clicks Save to… in Save dialog (or Load from… in Load dialog).
2
Picker opens at the current library location. Path input pre-filled.
3
User navigates: type/paste a path, click subfolders, or hit New folder…
4
Selected-folder bar updates as the user navigates — always shows what "Use this folder" will commit.
5
Click Use this folder → server persists libraryPath, picker closes.
6
Parent dialog's library row reflects the new path. Load: list refreshes. Save: primary button label flips to "Save to new folder" (per Section 5 dirty state).

Locked-in decisions

  • Library row labels: "Saved to" (Save dialog) / "Loaded from" (Load dialog).
  • "Open folder ↗" text link sits in the top-right of the path box, above the path itself.
  • Generic "Open folder" wording (no platform-specific Finder/Explorer split).
  • Autosave modal: head is the only message ("Auto-save is newer than the saved version") — no second "Replace current state?" question, since the user just clicked Load. The two options are rendered as standard preset-rows (same shape as the Load dialog list — bold filename + picks · workflows · saved-date) so nothing about the layout is new. Buttons collapse to NO / YES — load latest on one row.
  • Reset-to-default — dropped. Empty path inside the Save to… / Load from… picker clears the override if needed.
  • Recovery section: kind badge (Pre-load / Periodic) + timestamp + meta. No filenames exposed in rows.
  • Delete confirmation is inline in the command bar, not a separate modal. Target row gets a red outline; Close transforms to Yes.
  • Folder picker top has a short subtitle describing what the picker is for, matching every other dialog.
